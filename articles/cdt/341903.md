[非法加冯｜腾讯云：颜面尽失的草台班子](https://chinadigitaltimes.net/chinese/706672.html)
------

<p>昨天下午，2024年04月08日，腾讯云出现了一场全球性的大故障，用腾讯云官方的说法，崩了 74 分钟（15:31 – 16:45），波及全球 17 个区域与数十款服务。</p><div style="width:42%;float:right;padding-left:20px"><div class="su-spoiler su-spoiler-style-fancy href="https://chinadigitaltimes.net/chinese/copyright">详细版权说明</a>。</div></div></div><h2>事实影响是什么</h2><p><strong>但这与我观察到的事实不符</strong> —— 从故障范围上来说，这次的故障几乎是去年<a href="https://mp.weixin.qq.com/s?__biz=MzU5ODAyNTM5Ng==\&amp;mid=2247486452\&amp;idx=1\&amp;sn=29cff4ee30b90483bd0a4f0963876f28\&amp;scene=21#wechat_redirect" title="阿里云双十一史诗级大故障">阿里云双十一史诗级大故障</a>的翻版 —— 小道消息是整个管控面 GG，云 API 挂了，所以现象与去年阿里云如出一辙：依赖云 API 的云产品控制台不能用了。</p><p>被管控的纯资源，如云服务器 CVM，云数据库 RDS， 设置了公开读写访问对象存储 COS 不受影响可以继续使用。然而依赖认证与API 的各种云 PaaS 服务，例如标准的私有读写的对象存储 COS，就抓瞎了。</p><p>因为阿里云至今没有做一个像样的事后故障复盘，因此在《<a href="https://mp.weixin.qq.com/s?__biz=MzU5ODAyNTM5Ng==\&amp;mid=2247486468\&amp;idx=1\&amp;sn=7fead2b49f12bc2a2a94aae942403c22\&amp;scene=21#wechat_redirect" title="我们能从阿里云史诗级故障中学到什么">我们能从阿里云史诗级故障中学到什么</a>》中，我为阿里云的这次故障做了非官方的技术复盘。同样的判断逻辑完全也适用于这次故障 —— 这样的爆炸半径，根因出在 Auth 上的概率很大。目前，腾讯云仍然没有给出官方的事后故障复盘报告，也可能不会有了。</p><hr><h2>忽悠人的状态页</h2><p>我的朋友杨攀曾写过一篇《<a href="https://mp.weixin.qq.com/s?__biz=MjM5NzY0NzAwMQ==\&amp;mid=2456881916\&amp;idx=1\&amp;sn=7d62bfe73053c1f0460d318b2b87203e\&amp;scene=21#wechat_redirect" title="中国云服务走向全球？先把 Status Page 搞定">中国云服务走向全球？先把 Status Page 搞定</a>》，讨论了 Status Page （服务健康状态页）对于公有云服务的重要性，各家本土云厂商也跟进了这一特性，包括腾讯云。—— 状态页能在服务宕机的情况下有效减少客户的焦虑，降低沟通成本，但它的核心价值在于 “<strong>建立与客户的信任关系</strong>”。</p><p><img decoding="async" src="https://chinadigitaltimes.net/chinese/files/2024/04/post-706672-66155bbd979ee.png" alt="img"></p><p>看上去，腾讯云与阿里云的 Status Page 反应都比较迟缓，在故障发生后三四十分钟才开始更新。而不是像 <a href="http://mp.weixin.qq.com/s?__biz=MzU5ODAyNTM5Ng==\&amp;mid=2247487240\&amp;idx=1\&amp;sn=ba535fd0c1026bc2482ea6ad1e1fb8bf\&amp;chksm=fe4b3ad3c93cb3c50bfeaed64963cce25c49bee80364d3a8ca78b87d7c9f19fd4d79d3c62ddc\&amp;scene=21#wechat_redirect"><strong>Cloudflare</strong></a> 等产品一样及时更新故障，或采用自动化方式监测到故障后立即推送。但不同于阿里云 —— 虽慢却诚实地标记了所有服务受到影响，腾讯云的 Status Page 连基本的真实性与准确性都堪称稀烂。</p><p>例如，受到影响的对象存储 COS 服务，在有用户上报问题的几个可用区中，我并没有看到 Status 标红。而这样的例子还有更多。事实上如果问题真出在管控 API 上，那么影响的范围应该和阿里云一样 —— 所有服务的控制面。因此，这样鸡贼的做法只会给客户留下：“<strong>不透明、有猫腻</strong>“ 的负面印象。</p><hr><h2>撒谎的三无公告</h2><p>在故障出现 40 ～ 50 分钟后，腾讯云终于发出了第一份故障公告，也是截止到目前 Status Page 上唯一一份公告。但其内容就一句话 ——&nbsp;<strong>三无公告</strong>：无时间（故障时间），无地点（可用区/AZ），无范围（影响服务）。而且姗姗来迟，比我替它发的公告《<a href="https://mp.weixin.qq.com/s?__biz=MzU5ODAyNTM5Ng==\&amp;mid=2247487267\&amp;idx=1\&amp;sn=7d31d44e89560356b5c5a2e7a40bb1e1\&amp;scene=21#wechat_redirect" title="【腾讯】云计算史诗级二翻车来了">【腾讯】云计算史诗级二翻车来了</a>》还晚了十分钟。</p><p><img decoding="async" src="https://chinadigitaltimes.net/chinese/files/2024/04/post-706672-66155bbdb9cbd.png" alt="img"></p><p>但这份公告最致命的问题是真实性与准确性：首先，故障绝对不仅仅是“控制台”，而是整个控制面。作为一个专业的云计算服务供应商，一字之差天壤之别，混淆两者区别的原因，<strong>要么是蠢</strong>（缺乏专业素养，台面混为一谈）。<strong>要么是坏</strong>（避重就轻，推卸责任）。</p><p>请问，一个全身休克的人，说他 “面色异常”，这是一个真诚的回复吗？请问，一台被砸烂的笔记本电脑，说它“敲击键盘没有反应”是一个有意义的描述吗？同理，一个控制面爆炸的公有云，说自己“<strong>控制台异常</strong>”，是一个认真的回复吗？</p><p>其次，从事后官微的发布与用户群的反馈来看，在这个时间，<strong>“目前故障已恢复”</strong>&nbsp; <strong>是在撒谎</strong>。至少相当一部分服务的可用性事件是在 16:45 标记恢复的，在17 点前后，腾讯云产品吐槽群中也仍然有一些问题上报。</p><p><strong>我认为这份对腾讯云带来的伤害远比服务宕机要大的多</strong> —— 首先，在及时性，准确性上体现出了极差的专业素养。其次，在真实性上有意做手脚，会伤及公有云，或者说一切生意的根本 ——&nbsp;<strong>诚信</strong>。<strong>这对品牌形象是一个摧毁性打击。</strong></p><hr><h2>灾难级别的公关</h2><p>按理说，出现了这么严重的故障，应当用诚恳认真的态度去处理，但腾讯云官方微博居然还在抖机灵 —— <strong>堪称灾难级别的公关水平</strong>。</p><p><img decoding="async" src="https://chinadigitaltimes.net/chinese/files/2024/04/post-706672-66155bbddc413.png" alt="img"></p><p>这条微博也再次扇了腾讯云自己官网公告的大嘴巴子 —— 16:45 分发第一条帖子时，“<em><strong>工程师仍在紧急修复中</strong></em>”，17:16，距离第一次报告故障的 15:31已经过去近两个小时，“<em><strong>已经整体恢复</strong></em>”。然而，根据腾讯云官网 16:21 发布的公告[1]声称：“<em><strong>故障已恢复</strong></em>”。从实际情况来看，<strong>再次证明了官网公告在说谎</strong>。</p><p>阿里云双十一大故障的时候，刚刚开完云栖大会，打脸了吹下的极致高可用的牛逼，但毕竟隔了一周了。而腾讯云这次大故障的<strong>同时</strong>还在开发布会吹牛逼，还找特大号发了一篇软文：《<a href="https://mp.weixin.qq.com/s?__biz=MzI3MzAzNDAyMQ==\&amp;mid=2657846320\&amp;idx=1\&amp;sn=78ace56768acb5f2d8f914cc69687bd6\&amp;scene=21#wechat_redirect" title="太意外了！国内80%大模型都存在鹅厂！">太意外了！国内80%大模型都存在鹅厂！</a>》，发布时间 <strong>16:19</strong>，2分钟后官网发出故障通告，堪称光速打脸二次方。</p><p><img decoding="async" src="https://chinadigitaltimes.net/chinese/files/2024/04/post-706672-66155bbe09b8f." alt="img"></p><p>与之形成鲜明对照的是，<a href="http://mp.weixin.qq.com/s?__biz=MzU5ODAyNTM5Ng==\&amp;mid=2247486468\&amp;idx=1\&amp;sn=7fead2b49f12bc2a2a94aae942403c22\&amp;chksm=fe4b39dfc93cb0c92e5d4c67241de0519ae6a23ce6f07fe5411b95041accb69e5efb86a38150\&amp;scene=21#wechat_redirect">去年 11 月 Cloudflare 的故障</a>，Cloudflare CEO Matthew 亲自出来对故障进行道歉与复盘，相比之下，国内云厂商的危机公关堪称灾难级别 —— 彻底做实了草台班子的称号。</p><h2>实锤的草台班子</h2><p>请允许我引用瑞典马工的一句名言 ：“<a href="http://mp.weixin.qq.com/s?__biz=MzU5ODAyNTM5Ng==\&amp;mid=2247485363\&amp;idx=1\&amp;sn=8622b25fd2309d4fc969d22964a04129\&amp;chksm=fe4b3268c93cbb7efb8757e876574bebf5d615bdd3c4ceb76e48b2bd907d78faeb450ca1e825\&amp;scene=21#wechat_redirect"><strong>阿里云是个工程质量差劲的正经云，但腾讯云是一群业余销售加业务码农玩游戏</strong></a>”。所谓光鲜亮丽的大厂，在里面也不过是一个又一个的草台班子。</p><p><img
